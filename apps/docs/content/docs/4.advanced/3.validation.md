## Triggers

Triggers determine which interaction causes a field to validate. Configure them once on the form and override per field where needed.

| Trigger | Default | Fires when |
| --- | --- | --- |
| `onBlur` | `true` | Field loses focus |
| `onChange` | `true` | Value is committed |
| `onInput` | `true` | Every keystroke |
| `onFocus` | `false` | Field gains focus |
| `onMount` | `false` | Component mounts |

Toggle the buttons below to see how each trigger affects when validation fires.

::code-preview
  ::ValidationTriggersDemo
  ::

#code
```ts
const form = useNotForm({
  schema,
  validateOn: {
    onBlur: true,
    onInput: false,
    onChange: true,
    onFocus: false,
    onMount: false,
  },
})
```
::

### Per-field overrides

`validateOn` on `<NotField>` merges over the form config. Only the keys you specify are changed.

```vue
<!-- Form has onInput: false ŌĆö this field overrides it -->
<NotField path="username" :validateOn="{ onInput: true }" v-slot="{ events }">
  <input v-model="form.values.username" v-bind="events" />
</NotField>
```

```vue
<!-- Form has onBlur: true ŌĆö disable it for this field -->
<NotField path="tos" :validateOn="{ onBlur: false }" v-slot="{ events }">
  <input type="checkbox" v-model="form.values.tos" v-bind="events" />
</NotField>
```

## Modes

The mode controls re-validation behaviour after an error has first appeared. Type into each field, blur it, then keep typing to see the difference.

::code-preview
  ::ValidationModesDemo
  ::

#code
```ts
// Eager (default) ŌĆö re-validates on input while an error exists
const form = useNotForm({
  schema,
  validationMode: { eager: true },
})

// Lazy ŌĆö validates only on blur or submit
const form = useNotForm({
  schema,
  validationMode: { eager: false, lazy: true },
})
```
::

**`eager`** ŌĆö validates on blur, then re-validates on every input/change while an error exists. Use for short forms where instant feedback matters.

**`lazy`** ŌĆö validates only on blur or submit. No re-validation while typing. Use for long forms where constant feedback would feel intrusive.

## Field vs form validation

### `validateField`

Runs the full schema, then updates errors **only for the targeted path**. All other field errors are left untouched.

```ts
await form.validateField('email')
// Only errors for 'email' change
```

### `validate`

Runs the schema and replaces **all** errors with the result.

```ts
const result = await form.validate()

if (result.issues) {
  // Validation failed ŌĆö result.issues contains all errors
} else {
  console.log(result.value) // Typed, transformed schema output
}
```

## Async validation

All schema validation is awaited. Async rules in your schema ŌĆö HTTP calls, database lookups ŌĆö work without any extra configuration.

```vue
<NotField path="username" v-slot="{ events, isValidating }">
  <input v-model="form.values.username" v-bind="events" />
  <span v-if="isValidating">Checking availabilityŌĆ”</span>
</NotField>
```

## Cross-field validation

Cross-field rules belong in the schema. NotForm runs the full schema on every validation call, so `refine` and `superRefine` work correctly.

```ts
const schema = z.object({
  password: z.string().min(8),
  confirm: z.string(),
}).refine(v => v.password === v.confirm, {
  message: 'Passwords do not match',
  path: ['confirm'],
})
```

When the user blurs `confirm`, `validateField('confirm')` surfaces the cross-field error at that path. Other field errors are not touched.

## Server-side errors

Submit the form below with `taken@example.com` to see a server error applied to a specific field after submission.

::code-preview
  ::ServerErrorsDemo
  ::

#code
```ts
const form = useNotForm({
  schema,
  onSubmit: async (values) => {
    try {
      await $fetch('/api/register', { method: 'POST', body: values })
    } catch {
      form.setError({
        path: [{ key: 'email' }],
        message: 'This email is already registered',
      })
    }
  },
})
```
::

`setError` replaces an existing error for the same path, or appends it if none exists. To bulk-replace all errors at once (e.g., from a `422` response):

```ts
form.setErrors([
  { path: [{ key: 'email' }], message: 'Already taken' },
  { path: [{ key: 'username' }], message: 'Too short' },
])
```

::note
Server errors live in the same `errors` array as schema errors. When `validateField` runs again for a field, its server error is replaced by the schema result. To preserve a server error until the user actually changes the value, gate validation with `form.dirtyFields.has(path)`.
::
